**Phase A: Audit**

1. Start the dev server: `bun run build && bun run preview` (or use the dist output with a local server).
2. Run Lighthouse CLI on four page types (mobile preset, performance category):
```bash
npx lighthouse http://localhost:4321/ --preset=perf --only-categories=performance --emulated-form-factor=mobile --output=json --output-path=./lighthouse-home.json --chrome-flags="--headless=new"
npx lighthouse http://localhost:4321/shop --preset=perf --only-categories=performance --emulated-form-factor=mobile --output=json --output-path=./lighthouse-shop.json --chrome-flags="--headless=new"
npx lighthouse http://localhost:4321/shop/bowls --preset=perf --only-categories=performance --emulated-form-factor=mobile --output=json --output-path=./lighthouse-product.json --chrome-flags="--headless=new"
npx lighthouse http://localhost:4321/workshop --preset=perf --only-categories=performance --emulated-form-factor=mobile --output=json --output-path=./lighthouse-workshop.json --chrome-flags="--headless=new"
```
3. Extract scores from JSON: `jq '.categories.performance.score * 100' lighthouse-*.json`
4. If ALL scores >= 85, no fixes needed. Document scores and proceed to checkpoint.
5. If ANY score < 85, read the audit details to identify specific bottlenecks:
```bash
jq '[.audits | to_entries[] | select(.value.score != null and .value.score < 0.9) | {id: .key, title: .value.title, score: .value.score, displayValue: .value.displayValue}]' lighthouse-home.json
```

**Phase B: Fix (conditional — only if scores < 85)**

Apply fixes based on what the audit reveals. Likely candidates (from research):

- **GLightbox on product pages:** If product page score < 85 and the audit flags main-thread JS or render-blocking resources, evaluate removing GLightbox and replacing with a CSS-only zoom or `loading="lazy"` on the lightbox script. Per user decision: keep GLightbox if score is 85+ with it; replace only if it is the cause of failure.

- **CLS from scroll animations:** If CLS score is flagged, ensure animated elements (`.section-header, .product-card, .process-step, .insta-item, .reservation-step`) have explicit `min-height` or `aspect-ratio` so they don't shift when `fade-in` class is added. Per user decision: keep animations, fix dimensions.

- **Unused CSS:** If "reduce unused CSS" is flagged, check if the removed preloader CSS in global.css left orphaned rules (should be clean after Plan 01 keyframe removal, but verify).

- **Image sizing:** If "properly size images" is flagged, check if any `<Image>` component is missing width/height attributes or serving oversized images.

- **Font display:** Self-hosted fonts should already have `font-display: swap` via @fontsource. If audit flags font-related issues, verify the CSS.

After applying fixes, re-run Lighthouse to confirm scores are now >= 85.

**Phase C: Cleanup**
Delete the lighthouse-*.json output files (they are diagnostic artifacts, not project files).
